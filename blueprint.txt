Service Overview &amp; PhilosophyThe Auth Service is the identity backbone of LifeOps. Every request across all nine microservices traces back to a token issued here. It is the only service that stores passwords, issues JWTs, and manages user identity at the system level. In v2.0 it is also the subscription gateway — owning plan management, renewal tracking, and feature-access gating that powers the flexible subscription-based monetisation model.1.1  Core ResponsibilitiesIssue and validate JWT access tokens (15-minute expiry) and refresh tokens (30-day expiry)Store and manage user identity: email, username, hashed password, timezone, preferencesOwn the subscription lifecycle: plan selection, renewal dates, grace periods, cancellationManage feature-access gating: which services each subscriber tier can accessEnforce role-based access: USER, ADMIN, SUPER_ADMINPublish auth and subscription events to Redis for downstream servicesProvide a single internal HTTP endpoint other services call to validate tokens and check subscription access1.2  What This Service Does NOT DoIt does NOT store any domain data: no goals, no health logs, no transactionsIt does NOT make decisions about what a user can do inside a service — it only confirms who they are and what plan they are onIt does NOT send emails directly — it publishes events to the Notification ServiceIt does NOT handle OAuth (social login) in v1 — this is a v2 additionIt does NOT process payments directly — it records payment confirmation from the Payment Gateway and activates the subscription1.3  Technology StackPackageVersionPurposefastapi0.115+Async Python web framework. User Types &amp; Access ModelLifeOps uses a five-tier user model. Access is the dynamic intersection of role and active subscription plan. There is no wallet or per-use micro-deduction in v2.0 — access is determined entirely by the current subscription status.2.1  User Type DefinitionsTypeRole ValueWho They AreWhat Defines Their AccessGuest— (no account)Unauthenticated visitorNo JWT. Unlike binary monthly lock-ins, users can choose the commitment period that matches their financial rhythm — weekly, bi-weekly, monthly, quarterly, or annual.No wallet, no micro-deductions: A subscriber pays once for their chosen period and gets full access to their plan features until the period ends. No per-action charges.Free tier is real and permanent: A user who cannot afford any plan still gets meaningful value from LifeOps. The free tier is not a crippled demo.Grace periods prevent harsh cut-offs: If a subscription lapses (failed renewal, no top-up), the user gets a grace period before features are paused — never an abrupt lock-out.Flexible periods match African financial rhythms: Weekly and bi-weekly plans serve users paid on those cycles. Monthly, quarterly, and annual plans serve stable-income users who prefer to save with longer commitments.2.3  Plan Feature Access MatrixFeature / ServiceFreeBasicProMaxHealth Tracker✔ Unlimited✔ Unlimited✔ Unlimited✔ UnlimitedGoals Engine — manual✔ Up to 3 goals✔ Unlimited✔ Unlimited✔ UnlimitedGoals Engine — AI decomposition✗✔✔✔Money Pulse — manual logging✔ 2 envelopes✔ Unlimited✔ Unlimited✔ UnlimitedMoney Pulse — CSV import✗✔✔✔Relationship Map✔ Up to 10 contacts✔ Unlimited✔ Unlimited✔ UnlimitedGrowth Feed✔ 1 article/day✔ 3 articles/day✔ Unlimited✔ UnlimitedAI Review — weekly review✗✗✔✔AI Review — on-demand review✗✗✗✔Priority support✗✗✔✔Early access to new features✗✗✗✔3. Retry once at 24h and once at 48h.If still failed at 72h (end of grace): set status=EXPIRED. Publish subscription_expired event. User sees friendly downgrade message.On expiry, the JWT is NOT invalidated — the next token refresh picks up the updated subscription status and re-embeds it.3.5  Trial PolicyDuration: 7 days at Pro tier.Eligibility: One trial per account, ever. Tracked via user.trial_used boolean.No payment required to start: User selects a plan, clicks &quot;Start Free Trial&quot;, trial activates immediately.Conversion: At trial end, user is prompted to subscribe. If they do not, status moves to EXPIRED and they revert to Free.Grace period: 3 days post-trial to allow conversion without immediate feature loss.3.6  Upgrade &amp; Downgrade RulesUpgrade (e.g., Basic → Pro): immediate effect. Remaining days on current plan are credited proportionally (credit stored as balance_credit_rwf in the subscription record).Downgrade (e.g., Pro → Basic): takes effect at next renewal date. User keeps current plan features until then.Period change (e.g., weekly → monthly): takes effect at next renewal. No proration needed — user completes current period first.Cancellation: status set to CANCELLED. Features continue until current period ends, then EXPIRED.4. Security ArchitectureThis section describes the complete security stack — drawn from industry practices used by LinkedIn, Facebook/Meta, and other major platforms — implemented entirely with free, open-source packages.5.1  Cookie Strategy — HttpOnly + Secure + SameSiteLifeOps uses a two-token strategy identical to what major platforms use:TokenStorageExpiryWhyAccess Token (JWT)React state (memory only)15 minutesShort-lived. If stolen via XSS, expires quickly. NEVER in localStorage or sessionStorage.Refresh TokenHttpOnly cookie30 daysHttpOnly = JavaScript cannot read it. Cannot be stolen via XSS. Sent automatically by browser.⚡ IMPORTANTCRITICAL: The access token is stored ONLY in React component state (AuthContext). If the page is refreshed, the Shell&apos;s useEffect calls the refreshToken mutation on mount — the browser sends the HttpOnly cookie automatically, and the Auth Service validates it and returns a new access token. This is the same pattern used by Facebook and LinkedIn.Cookie Configuration (Python / FastAPI)response.set_cookie(    key=&apos;refresh_token&apos;,    value=raw_refresh_token,    httponly=True,    secure=True,    samesite=&apos;lax&apos;,    max_age=60 * 60 * 24 * 30,    path=&apos;/graphql&apos;,)5.2  JWT Strategy — RS256 Asymmetric SigningLifeOps uses RS256 (RSA + SHA-256 asymmetric signing). The Auth Service signs with a private key. Apollo Router and all services verify with the public key from the JWKS endpoint. The private key never leaves the Auth Service.JWT Payload Structure{  &apos;sub&apos;: &apos;user_mongodb_id&apos;,  &apos;email&apos;: &apos;user@example.com&apos;,  &apos;role&apos;: &apos;USER&apos;,  &apos;plan&apos;: &apos;PRO&apos;,             # FREE | BASIC | PRO | MAX  &apos;plan_expires&apos;: 1714592000, # Unix timestamp of current period end  &apos;iat&apos;: 1712000000,  &apos;exp&apos;: 1712000900,  &apos;jti&apos;: &apos;uuid4-unique-token-id&apos;,}ℹ INFOThe &apos;plan&apos; and &apos;plan_expires&apos; fields in the JWT mean services do not need to call the Auth Service to check subscription status — they read it directly from the token. The token refreshes every 15 minutes, so plan changes propagate within one token lifetime.5.3  CSRF ProtectionCSRF is automatically mitigated by the SameSite=Lax cookie setting. For additional hardness, LifeOps implements the Double-Submit Cookie pattern for mutations. On login the server sets a CSRF token cookie (NOT HttpOnly). The React Shell reads it and includes it in the X-CSRF-Token header of every GraphQL mutation. The Auth Service middleware validates that the header value matches the cookie value.5.4  Rate Limiting — slowapiEndpointLimitWindowRationalePOST /graphql (login)5 requests15 min per IP+emailBrute-force protection.POST /graphql (register)3 requests1 hour per IPPrevents mass account creation.POST /graphql (requestPasswordReset)3 requests1 hour per emailPrevents email bombing.POST /graphql (refreshToken)30 requests1 hour per userNormal usage is 4/hour.POST /graphql (subscribeToPlan)10 requests1 hour per userPrevents subscription abuse.All other authenticated endpoints200 requests1 min per userGeneral API abuse protection.5.5  Password Securitybcrypt cost factor 12 (~250ms — intentionally slow to resist brute force)Minimum 8 characters with at least 1 uppercase, 1 digit, 1 special characterMust not be in the top-10,000 common passwords list (loaded at startup)Must not contain the user&apos;s email prefix or username5.6  Security Headers — FastAPI MiddlewareX-Content-Type-Options: nosniffX-Frame-Options: DENYX-XSS-Protection: 1; mode=blockReferrer-Policy: strict-origin-when-cross-originPermissions-Policy: geolocation=(), microphone=()Strict-Transport-Security: max-age=63072000; includeSubDomains5.7  Token Rotation &amp; Refresh Token Reuse DetectionEach time a refresh token is exchanged for a new access token, the old refresh token is REVOKED and a new one is issued (refresh token rotation). If a revoked token is presented again, the Auth Service immediately revokes ALL tokens for that user — detecting theft. The legitimate user is prompted to log in again.6. Proxy Service (Nginx)A lightweight Nginx reverse proxy sits in front of the Auth Service (and eventually all LifeOps microservices). It handles TLS termination, pre-request rate limiting, CORS headers, and request routing — offloading these concerns from the FastAPI application itself.8.1  Why a Proxy Service?TLS termination: Nginx handles HTTPS so FastAPI can operate on plain HTTP inside the Docker network. This is the standard production pattern.Pre-filter rate limiting: Nginx&apos;s limit_req_zone blocks obvious abuse (e.g., DDoS) before the request even reaches Python. This protects the application tier.Static file serving: If the frontend is server-side rendered or static, Nginx can serve it directly without involving FastAPI.Single ingress point: All traffic enters the system through Nginx. Internal services communicate directly via Docker network without going through the proxy.Buffering &amp; timeouts: Nginx buffers slow client connections, protecting FastAPI from slow-loris attacks.8.2  Nginx Configuration# nginx/nginx.confworker_processes auto;events { worker_connections 1024; Key Authentication &amp; Subscription Flows9.1  Registration FlowUser submits: email, username, password, timezone.Validate email format (Pydantic EmailStr). Normalize to lowercase.Check email uniqueness. Return CONFLICT if taken.Check username uniqueness. Return CONFLICT if taken.Validate password strength.Hash password: bcrypt cost 12.Create user document with role=USER, is_active=True, email_verified=False, current_plan=FREE.Generate email verification token (itsdangerous, 24h). Store SHA-256 hash.Publish user_registered event to Redis.Return AuthPayload with access token + set HttpOnly refresh token cookie.9.2  Login FlowCheck if IP+email is rate-limited (Redis counter). If so, return 429 with retry_after.Look up user by email. Run dummy bcrypt check if not found (prevents timing attacks).Check is_active=True and email_verified=True.Check locked_until. If in future, return 423.Verify password. On failure: increment failed_login_attempts. Lock after 5 failures.On success: reset failed_login_attempts=0, update last_login_at.Fetch active subscription to embed plan and plan_expires in JWT.Generate RS256 JWT with sub, email, role, plan, plan_expires, iat, exp, jti.Generate refresh token, hash, store. Set HttpOnly cookie + CSRF cookie.Write LOGIN_SUCCESS audit log. Return AuthPayload.9.3  Token Refresh FlowRead refresh_token from HttpOnly cookie. If absent, return 401.Hash the token, look up in refresh_tokens.If revoked_at is set: REUSE DETECTED. Revoke ALL tokens. Return 401 SESSION_COMPROMISED.If expired or user inactive: return 401.Re-fetch active subscription to embed current plan into new JWT (handles plan changes between refreshes).Mark old token as revoked. Issue new JWT and new refresh token.Write TOKEN_REFRESHED audit log. Return new AuthPayload.9.4  Subscribe to Plan FlowUser selects a plan and billing period in the frontend, completes payment via Payment Gateway.Payment Gateway calls a webhook OR the frontend confirms payment and calls subscribeToPlan mutation with the payment_reference.Auth Service validates the payment_reference against the Payment Gateway API (or trusts a signed webhook).Check if user already has an active subscription. If upgrading, calculate proration credit.Create new Subscription document with status=ACTIVE, period_start=now, period_end=now+period, price_paid_rwf=price.Update user.current_plan to the new plan (denormalised for fast JWT embedding).If trial_used=False and this is the first subscription: no trial consumption (trial is opt-in via startTrial mutation).Write SUBSCRIPTION_STARTED audit log.Publish subscription_started event to Redis.Return the new SubscriptionType object.9.5  Subscription Expiry FlowA background job (APScheduler or a dedicated cron container) runs every hour.It queries: subscriptions where period_end &lt; now and status=ACTIVE.For each: if auto_renew=True, attempt renewal via Payment Gateway. On success: create new subscription record with status=ACTIVE, period_start=old period_end.On payment failure: set status=GRACE, grace_end=period_end+3days. Publish subscription_grace_period_started event.A second pass queries: subscriptions where grace_end &lt; now and status=GRACE.For each: set status=EXPIRED. Update user.current_plan=FREE. Write SUBSCRIPTION_EXPIRED audit log. Publish subscription_expired event.User&apos;s next token refresh will pick up current_plan=FREE automatically.9.6  Trial FlowUser calls startTrial mutation.Check trial_used=False. If already used: return error TRIAL_ALREADY_USED.Create Subscription: plan=PRO, billing_period=WEEKLY (7 days), status=TRIAL, price_paid_rwf=0.Set user.trial_used=True, user.current_plan=PRO.Write TRIAL_STARTED audit log. Publish trial_started event.At trial end: same expiry flow as above. Grace period of 3 days to convert.10. Graceful Degradation &amp; Subscription Expiry PhilosophyWhen a subscription expires or lapses, the system must respond with dignity and clarity — never with a confusing error or a shame-inducing message. This principle is non-negotiable for the LifeOps market positioning.✔ NOTEThe free tier is real and meaningful. A user who cannot afford a subscription still gets genuine value from LifeOps. Subscription expiry should feel like a gentle step down, not a punishment.19.1  What All Users Can Do (Always Free, No Subscription Required)Log in and access their accountUse the Health Tracker (unlimited — always free)View and manage up to 3 goals manually (no AI decomposition)Use Money Pulse with up to 2 envelopes (no CSV import)Manage up to 10 contacts in the Relationship MapRead 1 Growth Feed article per dayAccess the Dashboard with current dataManage their subscription and billing from the app19.2  What Happens When a Subscription ExpiresDuring grace period (3 days after period end): User retains full plan features. A persistent banner is shown: &quot;Your subscription ended on [date]. Renew to keep your premium features.&quot;After grace period expires: user.current_plan is set to FREE. The next token refresh embeds plan=FREE. Premium features are gracefully hidden, not broken.
